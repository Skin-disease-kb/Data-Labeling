# 皮肤病诊断便携式产品技术方案

## 一、项目概述

基于现有的皮肤病图像标注工具，构建一个完整的便携式皮肤病诊断系统。系统采用"边缘计算+安全防护"的架构，将AI诊断能力部署到便携式硬件（香橙派）上，实现本地化、隐私保护的皮肤病初步筛查。

**核心目标**：
- 利用 Qwen3-VL 视觉语言模型实现皮肤病图像分类
- 集成安全防护模型过滤恶意输入
- 部署到香橙派实现便携式、离线运行

---

## 二、系统架构设计

### 2.1 整体架构

系统采用"三段式"架构，从用户输入到诊断输出经过完整的处理流程：

```
用户（手机/电脑）
        ↓
    ┌──────────────────────┐
    │  Web 前端界面   │
    └────────┬─────────────┘
              │
              ↓（HTTP请求）
    ┌──────────────────────┐
    │   Orange Pi 边缘设备   │
    │                      │
    │  ┌────────────────┐   │
    │  │  安全防护层   │   │
    │  │ Qwen3Guard    │   │
    │  └──────┬────────┘   │
    │         │ 安全检查    │
    │         ↓ 通过后      │
    │  ┌────────────────┐   │
    │  │  诊断推理层   │   │
    │  │ Qwen3-VL-8B   │   │
    │  └──────┬────────┘   │
    │         │          │
    └─────────┴──────────┘
              ↓（JSON响应）
         用户看到诊断结果
```

### 2.2 各层功能

**Web 前端层**
- 提供友好的用户界面，支持拖拽上传图像
- 显示诊断进度和结果
- 展示医疗免责声明（强调"仅供参考"）

**安全防护层**
- 在用户输入到达诊断模型前进行检查
- 拦截提示词注入、角色扮演攻击等恶意输入
- 只有安全的请求才会传递给诊断模型

**诊断推理层**
- 接收安全检查通过后的图像和文本
- 调用微调后的 Qwen3-VL-8B 模型进行分类
- 输出5种皮肤病类型的预测结果

### 2.3 数据流

1. 用户通过浏览器访问部署好的Web页面
2. 上传皮肤病变图像（可选附文字描述）
3. Orange Pi 上的服务接收请求
4. 先经过 Qwen3Guard 安全检查（输入是否恶意、是否适合医疗场景）
5. 安全通过后，调用 Qwen3-VL-8B 模型分析图像
6. 模型输出疾病类别（中文名称+英文名称+置信度）
7. Web 页面展示结果，同时显示免责声明

## 三、模型选择

### 3.1 主诊断模型：Qwen3-VL-8B-Instruct

**选择理由**：
- Qwen3-VL 是阿里巴巴最新发布的视觉语言模型，在多模态理解方面表现出色
- 8B参数规模在准确率和推理速度之间有良好平衡
- 支持4-bit量化，适合边缘设备部署
- 已有成熟的推理框架（llama.cpp）支持

**模型特点**：
- 具备视觉编码器，能理解图像特征
- 具备语言解码器，能输出结构化的分类结果
- 支持中英文双语，适合国内用户

**量化策略**：
- 使用4-bit量化将模型大小从约16GB压缩到约4-5GB
- AWQ（激活感知量化）或 GGUF 格式
- 推理精度损失小于2%，但内存占用减少75%

### 3.2 安全防护模型：Qwen3Guard-Gen-4B

**选择理由**：
- 官方安全模型，专门为Qwen系列设计
- 4B参数比8B版本更快，适合边缘部署预处理场景
- 三级分类系统（Safe/Controversial/Unsafe）比简单的二分类更灵活

**防护能力**：
- 提示词注入攻击检测
- 越狱尝试识别
- 不当内容过滤
- 医疗场景适配

**集成模式**：
采用**预处理模式**（Pre-processing），即在用户输入到达主模型前先进行安全检查。

优点：
- 节省计算资源（恶意输入不会消耗主模型推理）
- 响应更快（安全检查只需几百毫秒）
- 适合批量处理场景

缺点：
- 无法检测模型生成过程中的风险（但对于医疗诊断场景影响较小）



## 四、模型微调方法

### 4.1 微调目标

将通用的 Qwen3-VL-8B-Instruct 模型，通过少量标注数据训练，使其能够准确识别5种特定皮肤病类型：

| 疾病类别 | 英文名称 | 样本数量 | 占比 |
|----------|---------|----------|------|
| 光化性角化病 | Actinic Keratosis (ACK) | 730 | 55% |
| 脂溢性角化病 | Seborrheic Keratosis (SEK) | 235 | 18% |
| 鳞状细胞癌 | Squamous Cell Carcinoma (SCC) | 192 | 14% |
| 皮肤纤维瘤 | Dermatofibroma (DF) | 115 | 9% |
| 黑色素瘤 | Melanoma (MEL) | 52 | 4% |

### 4.2 微调技术：LoRA（低秩适应）

**为什么不全量微调？**
- 全量微调需要更新所有8B个参数，存储需求巨大（16GB+）
- 训练时间长，需要强大的GPU
- 容易过拟合

**LoRA的优势**：
- 只训练约1%的参数（约5000万 vs 80亿）
- 存储需求小（LoRA适配器仅几百MB）
- 训练速度快，普通显卡即可完成
- 可以方便地"插拔"不同的适配器

### 4.3 LoRA工作原理

```
原始模型（冻结）┐
                │
                ├─ LoRA A矩阵（可训练）
                │
                └─ LoRA B矩阵（可训练）
                       ↓
                 W_new = W_original + B × A
                       ↓
                    微调后的模型
```

实际训练中，原始模型的8B个参数保持不变，只训练两个低秩矩阵A和B（例如秩为16）。推理时，将BA矩阵加到原始权重上即可。

### 4.4 数据准备

**数据格式转换**：
将标注工具输出的CSV格式转换为模型训练需要的对话格式。

原始数据是文件夹结构（如 `ACK_光化性角化病`），说明数据已经按类别组织好了。我们只需要：
1. 从文件夹名称提取类别标签
2. 将每张图像组织成对话格式：
   - 用户："分析这张图片"
   - 助手："光化性角化病"

**数据增强（解决类别不平衡）**：
- MEL（黑色素瘤）只有52张，占比只有4%，直接训练会导致模型不重视这个类别
- 通过旋转、翻转、调整亮度等方式扩充少数类样本
- 对MEL和DF进行3-4倍过采样
- 训练时设置类别权重，让模型更关注少数类

**数据划分**：
- 训练集80%，验证集10%，测试集10%
- 采用分层采样，确保每个类别在三个子集中都保持原始比例

### 4.5 微调配置

**关键超参数**：
- 学习率：1e-4（比通用任务稍低，医疗场景需要更稳定）
- 批量大小：4（受显存限制）
- 训练轮数：3（完整训练3个epoch）
- LoRA秩（r）：16（秩越大，参数越多，但可能过拟合）
- 梯度累积：2（有效批量=8，提高训练稳定性）

**早停机制**：
- 监控验证集的F1分数
- 如果3个epoch内F1不提升，提前停止训练
- 防止过拟合

**损失函数**：
- 使用交叉熵损失作为主损失
- 结合类别权重（少数类损失乘以更大权重）

### 4.6 微调后的模型

**保存形式**：
- LoRA适配器（几百MB）：用于部署和更新
- 合并后的完整模型（约5GB 4-bit）：用于最终推理

**推理效果**：
- 预期准确率：85-90%（取决于数据质量和增强效果）
- 黑色素瘤（MEL）召回率目标：>=75%（最重要，因为漏诊风险大）

## 五、部署方案

### 5.1 Orange Pi 上的软件架构

```
┌────────────────────────────────────┐
│        操作系统                │
│  (Ubuntu/Debian ARM64)      │
└──────────────┬─────────────────┘
               │
    ┌──────────┴──────────────┐
    │   Python 虚拟环境         │
    │  (FastAPI + llama.cpp)    │
    └──────────┬───────────────┘
               │
      ┌─────────┴────────────┐
      │  模型内存加载区       │
      ├─ 安全模型 (~1.5GB)    │
      └─ 主模型 (~5GB)       │
              │
              ↓
         推理执行 (NPU)
```

### 5.2 服务启动流程

1. Orange Pi 上电，系统启动
2. systemd 自动启动 Python 服务
3. 服务加载 Qwen3Guard 安全模型（小模型，快速加载）
4. 主模型延迟加载（有请求时才加载，节省内存）
5. Nginx 提供静态Web页面，反向代理API请求
6. 用户访问，完成诊断

### 5.3 性能预期

**单次诊断耗时**：
- 安全检查：~300-500ms
- 图像预处理：~100ms
- 主模型推理：~2-2.5秒
- 响应传输：~100ms
- **总计：约3秒**

**并发能力**：
- 内存可容纳约2-3个并发请求
- 超出则排队处理

## 六、医疗安全与合规

### 6.1 免责声明（必须）

产品必须明确标注：
```
⚠️ 重要提示

本系统提供的诊断结果仅供参考，不能替代专业医生的诊断。

1. 本结果基于AI模型分析，可能存在误差
2. 如有疑虑或症状持续，请及时就医
3. 本系统不对误诊或漏诊承担责任
4. 紧急情况请直接拨打急救电话
```

### 6.2 隐私保护

- 所有处理在本地设备完成，图像不上传到云端
- 推理完成后立即释放内存
- 日志不保存原始图像（可选：保存匿名化的诊断记录）

### 6.3 使用限制

- 仅适用于皮肤表面的可见病变
- 不能诊断内部器官疾病
- 不能替代病理检查或活检
- 建议作为初步筛查工具

## 九、后续扩展方向

如果第一阶段成功，可以考虑：

1. **模型升级**：当更多标注数据积累后，用更大模型（如Qwen3-VL-32B）或更高精度版本
2. **多模态输入**：支持语音输入，方便老年人使用
3. **历史记录**：本地保存匿名化诊断记录，生成趋势报告
4. **远程问诊**：结合蓝牙设备连接医院系统，可疑病例自动推送
5. **健康数据**：集成血压、血糖等基础健康数据，提供综合分析
